# Fedi-Xanadu 运营发展规划：内容冷启动 + 社媒增长

> 内部文档，补充商业计划书第五章「增长策略」

## 一、战略定位：成为更好的 csdiy.wiki

[csdiy.wiki](https://csdiy.wiki) 的成功证明了一件事：**理工科学习者对结构化、高质量、免费/低成本学习路径有巨大需求**。但 csdiy.wiki 有明显的天花板：

| csdiy.wiki 的局限 | Fedi-Xanadu 的解法 |
|---|---|
| 纯静态站点，只是链接聚合 | 内容原生托管，Typst 排版，公式/图表一流体验 |
| 单一维护者，内容更新慢 | Fork 机制 + 社区投票，内容持续演化 |
| 无结构化依赖关系 | 技能树 DAG + 前置知识声明，精准匹配学习者水平 |
| 仅覆盖 CS | 四大领域：数学、物理、CS、经济学 |
| 无社区互动 | 评论、投票、关注、收藏、协作 |
| 无付费机制，创作者无收益 | 付费专栏 + AI Token，创作者可持续变现 |
| 无版本管理 | Pijul VCS 驱动，内容可追溯、可分叉 |

**一句话：csdiy.wiki 是一个人整理的课程书签列表；Fedi-Xanadu 是一个社区驱动的、有版本控制的、可付费的结构化知识平台。**

## 二、CC 内容冷启动方案

### 2.1 核心思路

平台最大的冷启动难题是**内容空**。解法：**大规模导入 Creative Commons 协议的优质学术内容，让用户一上来就有东西可看、可学、可 Fork**。

这不是抄袭——CC 协议明确允许再分发和改编，只要遵守署名等条款。平台成为 CC 内容的最佳阅读/学习界面，同时吸引原作者入驻。

### 2.2 内容源清单

#### 第一优先级：教材与课程资料

| 来源 | 内容类型 | 协议 | 导入策略 |
|------|----------|------|----------|
| **MIT OpenCourseWare** | 课程讲义、习题、考试 | CC BY-NC-SA | 按课程组织为系列，标注前置知识 |
| **OpenStax** | 完整教材（微积分、大学物理、经济学原理等） | CC BY | 按章节拆分为文章，组织为系列 |
| **OSSU (Open Source Society University)** | CS 课程路径 | CC BY-SA | 导入为技能树结构 |
| **Mathematics LibreTexts** | 数学教材库 | CC BY-NC-SA | 按主题导入，建立数学技能树 |
| **Physics LibreTexts** | 物理教材库 | CC BY-NC-SA | 同上 |
| **Runestone Academy** | 交互式 CS 教材 | 多为 CC | 提取文本内容，标注交互部分 |

#### 第二优先级：论文与学术资源

| 来源 | 内容类型 | 协议 | 导入策略 |
|------|----------|------|----------|
| **arXiv** | 预印本论文（数学、物理、CS、经济学） | 多为 CC BY / CC0 | 筛选高引用入门级综述（survey），标注前置知识 |
| **PLOS ONE / PLOS Computational Biology** | 同行评审论文 | CC BY | 筛选教学性质强的论文 |
| **Journal of Open Source Education (JOSE)** | 开源教育论文 | CC BY | 直接导入 |
| **Distill.pub 存档** | 高质量 ML 可视化文章 | CC BY | 高优先级导入，这类内容极其稀缺 |

#### 第三优先级：博客与社区内容

| 来源 | 内容类型 | 协议 | 导入策略 |
|------|----------|------|----------|
| **3Blue1Brown 文字版** | 数学可视化讲解 | 需确认协议 | 如有 CC 协议则导入 |
| **Brilliant.org 社区 wiki** | 数学/科学词条 | CC BY-SA | 作为知识节点补充技能树 |
| **nLab** | 高等数学 wiki | CC BY-SA | 筛选入门级内容 |
| **HyperPhysics** | 物理概念库 | 需确认 | 如允许则作为物理技能树基础 |
| **OER Commons** | 综合开放教育资源 | 多种 CC | 按领域筛选 |

### 2.3 导入工作流

```
发现内容 → 验证协议 → 格式转换 → 标注元数据 → 导入平台 → 社区优化
```

**详细步骤：**

1. **协议验证（法务合规）**
   - 逐源确认 CC 协议版本和具体条款
   - CC BY：署名即可自由使用（最宽松）
   - CC BY-SA：署名 + 衍生作品需同协议（兼容我们的 AGPL）
   - CC BY-NC-SA：署名 + 非商业 + 同协议（**注意：付费专栏中不可使用此类内容**，但免费内容可以）
   - CC BY-NC：署名 + 非商业（同上限制）
   - 建立协议兼容性矩阵，标记哪些内容可用于付费场景

2. **格式转换（技术实现）**
   - LaTeX → Typst 自动转换（编写转换脚本/利用 pandoc）
   - HTML → Typst（针对 LibreTexts 等网页内容）
   - PDF → Typst（针对无源码的讲义，可借助 AI 辅助）
   - Markdown → Typst（最简单的情况）
   - 数学公式重点验证，确保转换后渲染正确

3. **元数据标注（核心价值所在）**
   - **前置知识声明**：每篇内容标注需要的技能树节点（如：「本文需要：线性代数基础、概率论基础」）
   - **难度等级**：入门 / 进阶 / 高级
   - **所属领域**：数学 / 物理 / CS / 经济学
   - **系列组织**：将同一课程的内容组织为有序系列
   - 这是我们相比原始 CC 内容的**核心增值**——结构化

4. **导入与署名**
   - 以「Fedi-Xanadu 官方」账号发布，明确标注原作者和协议
   - 每篇文章顶部显示：`本文基于 [原作者] 的作品，采用 CC BY-SA 4.0 协议`
   - 保留原文链接，鼓励用户访问原始来源

5. **社区优化循环**
   - 导入的内容作为「种子版本」，社区可以 Fork 改进
   - 翻译（英→中）是最直接的改进方向
   - 纠错、补充例题、添加可视化
   - 投票机制筛选最优版本

### 2.4 分阶段执行计划

**第一阶段（第 1-2 月）：核心课程覆盖**

目标：导入 4 个领域各 1-2 门核心基础课，共约 200 篇内容

| 领域 | 课程 | 来源 | 预计文章数 |
|------|------|------|------------|
| 数学 | 微积分 | OpenStax *Calculus* | ~50 |
| 数学 | 线性代数 | MIT OCW 18.06 讲义 | ~30 |
| 物理 | 大学物理（力学+电磁学） | OpenStax *University Physics* | ~40 |
| CS | 数据结构与算法 | Runestone / Open DSA | ~40 |
| 经济学 | 微观经济学原理 | OpenStax *Principles of Economics* | ~40 |

**第二阶段（第 3-4 月）：深度扩展**

目标：每个领域扩展到 5-8 门课程，共约 1000 篇内容

- 数学：概率论、实分析、抽象代数、微分方程
- 物理：量子力学、统计力学、相对论入门
- CS：操作系统、计算机网络、数据库、编译原理、机器学习
- 经济学：宏观经济学、博弈论、计量经济学

**第三阶段（第 5-6 月）：论文与前沿内容**

目标：导入高质量综述论文和前沿教程

- arXiv 高引用综述（每领域 20-30 篇）
- Distill.pub 全站存档
- JOSE 精选教育论文

### 2.5 技术工具开发

为支撑大规模导入，需要开发以下工具：

| 工具 | 功能 | 优先级 |
|------|------|--------|
| `fx-import-latex` | LaTeX → Typst 批量转换 | P0 |
| `fx-import-html` | HTML → Typst 批量转换（含公式提取） | P0 |
| `fx-import-metadata` | AI 辅助标注前置知识和难度 | P1 |
| `fx-import-openstax` | OpenStax API 对接，自动拉取教材 | P1 |
| `fx-import-arxiv` | arXiv API 对接，筛选 CC 论文 | P2 |

### 2.6 法律与伦理注意事项

- **严格遵守 CC 协议条款**，尤其是署名（BY）和非商业（NC）要求
- NC 内容**绝不**出现在付费专栏中，仅作为免费公共内容
- 建立清晰的版权标注系统，让用户一眼能看到每篇内容的协议
- 如果原作者联系我们要求移除，**立即配合**（虽然 CC 协议不可撤回，但维护关系比内容重要）
- 长期目标：吸引原作者本人入驻，将导入内容「认领」为自己的账号

## 三、增长策略：以技术交流代替「做自媒体」

核心原则：**两个人的团队不应该去当自媒体博主**。所有对外输出都应该是「我在做一件有趣的事，顺便聊聊」，而不是「请来用我的产品」。技术深度内容本身就是最好的获客手段。

### 3.1 长文输出（知乎 / 即刻）

写 1-2 篇有技术深度的长文，这种内容的受众和我们的目标用户高度重合：

**选题方向：**

| 选题 | 切入角度 | 为什么有效 |
|------|----------|------------|
| 「为什么我用 Pijul 做知识版本控制」 | Patch theory vs snapshot-based VCS，为什么 Git 的合并模型不适合文档协作，Pijul 的交换律如何让 Fork 变得优雅 | Pijul 在中文互联网几乎零讨论，这篇文章会成为搜索结果第一页的常驻内容 |
| 「技能树匹配：一种新的内容分发思路」 | 知乎/推荐算法的问题是「猜你喜欢」，而学习内容的核心问题是「你够不够格看懂」。用 DAG 建模前置知识依赖，按技能树匹配内容 | 这是产品层面的创新叙事，不是技术炫技，容易引发讨论 |
| 「AT Protocol 实战：用去中心化身份做知识平台」 | Bluesky 的账号体系怎么用、为什么选 AT Protocol 而不是 ActivityPub、DID 带来的数据自主权 | AT Protocol 话题热度正高，蹭得上 |

**发布策略：**
- 知乎发完整版，即刻发精简版 + 讨论
- 文章结尾自然提一句：「这是我正在做的开源项目 Fedi-Xanadu，欢迎试用/贡献」
- 不需要高频更新，一篇高质量长文的长尾流量远超 10 篇水文

### 3.2 Show HN / Lobste.rs（英文技术社区）

**为什么值得做：**
- 开源 Rust 项目 + AT Protocol + Pijul 这个组合在英文社区极度稀缺
- Hacker News 的一个热帖可以带来数百个 GitHub star 和大量技术讨论
- 国际曝光对后续融资叙事也有帮助（「项目在 HN 上获得了关注」）

**执行要点：**

| 要素 | 建议 |
|------|------|
| 标题格式 | `Show HN: Fedi-Xanadu – A knowledge platform built with Rust, Pijul, and AT Protocol` |
| 正文 | 简短说明动机（理工科内容平台的痛点）、技术选型理由、当前状态、在线 Demo 链接 |
| 时机 | 美国西海岸上午（北京时间凌晨），工作日周二-周四 |
| 准备 | 提前确保 Demo 站点稳定，README 写好英文版，准备好回复评论 |

同步发到 **Lobste.rs**（需要邀请码，可以在相关技术社区找人邀请）。

### 3.3 高校技术分享

**定位：技术交流，不是产品推广。**

去华科或其他高校的 CS 社团/开源社区做一次技术分享，话题选择：

| 话题 | 受众 | 自然引出项目的方式 |
|------|------|---------------------|
| Patch Theory 与 Pijul：超越 Git 的版本控制 | CS 学生、PL 爱好者 | 「我在自己的项目里实践了 Pijul，效果是这样的...」 |
| AT Protocol 实战：去中心化社交协议入门 | Web 开发者、分布式系统爱好者 | 「我用 AT Protocol 做了一个知识平台的登录系统...」 |
| Typst：下一代学术排版系统 | 写论文的研究生 | 「我把 Typst 集成到了一个在线平台里，支持实时渲染...」 |

**关键：** 分享的 80% 是纯技术内容（对听众有价值），20% 是项目实践（自然带出）。听众感兴趣会自己去搜项目，不需要你推。

**可执行路径：**
- 联系华科开源俱乐部 / LUG / ACM 协会
- 如果没有合适的线下机会，可以先在 Rust 中文社区的线上活动做一次 lightning talk

### 3.4 B站技术视频

**不需要做「UP主」，做 1-2 个有深度的视频就够了：**

| 视频类型 | 内容 | 目的 |
|----------|------|------|
| 项目 Demo | 3-5 分钟，演示完整工作流：Typst 写作 → 技能树匹配 → Fork → 投票 | 让人直观看到产品形态 |
| 技术讲解 | 10-15 分钟，讲 Pijul 或 AT Protocol 的某个有趣细节 | 长尾搜索流量，B站技术视频生命周期很长 |

B站技术视频的长尾效应极强——一个讲 Pijul 的视频可能半年后还在被搜到，因为中文互联网上讲这个的人几乎没有。

### 3.5 小红书

小红书的用户基数大，理工科学习类内容有受众。同样的原则——内容本身有价值，不需要刻意运营人设：

| 内容类型 | 示例 | 为什么有效 |
|----------|------|------------|
| 技能树截图 | 「数学从高中到研究生的完整技能树」 | 天然视觉素材，小红书用户爱收藏路径图 |
| Typst 排版对比 | 「LaTeX vs Typst 排出来的笔记对比」 | 视觉冲击力强，能引发讨论 |
| 学习路径帖 | 「自学线性代数的前置知识链，你可能跳过了第二步」 | 实用价值高，技能树理念的自然载体 |
| 项目思考 | 「两个大学生做了一个开源知识平台」 | 小红书上创业/独立开发叙事有流量 |

不用追求频率，有值得分享的东西就发。

### 3.6 增长预期

这种策略不会带来爆发式增长，但会带来**高质量用户**：

| 渠道 | 预期效果 | 用户质量 |
|------|----------|----------|
| 知乎长文 | 单篇 5,000-50,000 阅读（取决于话题热度） | 高——能读完技术长文的人就是目标用户 |
| Show HN | 热帖可获 200-500 GitHub star，100+ 注册 | 极高——技术社区早期用户，可能变成贡献者 |
| 高校分享 | 现场 30-100 人，后续口碑传播 | 高——直接触达校园种子用户 |
| B站视频 | 首月 1,000-5,000 播放，长尾持续 | 中高——技术视频观众精准 |
| 小红书 | 视觉素材可能意外走红 | 中——需要进一步筛选转化 |

## 四、与 csdiy.wiki 的差异化竞争策略

### 4.1 不是替代，是超集

csdiy.wiki 解决了「学什么」的问题，但没解决：
- **在哪学**（它只给链接，用户要跳转到 N 个不同网站）
- **怎么学**（没有前置知识依赖，不知道自己够不够格学）
- **学了之后呢**（没有社区，没有讨论，没有协作）

Fedi-Xanadu 的定位是：**csdiy.wiki 的内容 + 知乎的社区 + 小报童的付费 + GitHub 的协作，统一在一个有技能树的平台上**。

### 4.2 具体策略

1. **导入 csdiy.wiki 的课程结构**（它本身是 CC BY-SA）
   - 将其推荐的课程路径导入为技能树
   - 将课程对应的 CC 内容直接托管在平台上
   - 用户不再需要跳转，一站式完成学习

2. **扩展到 csdiy.wiki 不覆盖的领域**
   - 数学、物理、经济学——这三个领域目前没有同等质量的中文学习路径
   - 先做好这三个领域，形成差异化

3. **社区驱动 vs 个人维护**
   - csdiy.wiki 依赖 PKUFlyingPig 一人维护，更新频率受限
   - Fedi-Xanadu 的 Fork + 投票机制让社区共同维护，内容演化更快

4. **中文化**
   - csdiy.wiki 推荐的大多是英文资源
   - 我们通过 AI 翻译 + 社区翻译，提供中英双语版本
   - 对中国学生来说，中文优质内容是刚需

## 六、风险补充

| 风险 | 应对 |
|------|------|
| CC-NC 内容误用于付费场景 | 建立协议标签系统，付费专栏提交时自动检查内容协议兼容性 |
| 大量导入内容质量参差不齐 | 优先导入知名教材（OpenStax 等），劣质内容靠社区投票下沉 |
| 转换工具 LaTeX→Typst 精度不够 | 先人工校验首批内容，积累错误模式后改进转换器 |
| 社媒运营人力不足（2 人团队） | 知乎内容可复用平台文章，小红书用模板化生产降低成本 |
| 被指责「搬运」CC 内容 | 显著署名 + 主动联系原作者 + 强调增值（结构化、翻译、技能树） |
